# 平台架构

## 设计目标

平台只拥有跨游戏事实：账号、身份关联、会话、游戏目录与通用积分。游戏服务器继续拥有游戏规则、房间、世界和局内状态，防止平台模块演化为不可维护的全能服务。

## 身份模型

`platform_accounts` 是账号主体，`platform_identity_links` 将 `local`、`dst-platform` 等身份提供者的稳定 subject 映射到同一个 `accountId`。用户名不能作为跨平台合并依据，账号关联必须基于已验证的 provider subject。

本地密码使用带随机盐的 scrypt。浏览器只保存 HttpOnly 会话 Cookie，不接触长期访问令牌。

## 游戏启动

1. 大厅调用 `POST /api/v1/games/:gameId/launch`。
2. 平台生成 60 秒有效的一次性启动码，只保存哈希。
3. 浏览器导航到游戏并携带 `launch_code`。
4. 游戏服务端使用自己的服务令牌向平台兑换可信 `GamePrincipal`。
5. 游戏服务端签发限定当前游戏的 HttpOnly 会话 Cookie。
6. Socket.IO 中间件在建立连接前验证游戏会话，业务事件不再接收浏览器自报身份。

## 通用积分

积分余额由不可变流水推导并缓存。游戏服务端必须提交 `gameId + idempotencyKey`，数据库先按幂等键取得事务锁：完全相同的重复请求返回原流水，参数不一致则明确拒绝；消费前锁定钱包并阻止负余额。游戏局内能量、赛季积分和胜负数据不属于通用积分。

## 故障边界

- 平台暂时不可用时，已经建立的短期游戏会话可以继续运行。
- 游戏不可用不影响大厅账号和其他游戏。
- 积分写入失败时游戏应保留幂等事件并重试，不能由浏览器补发奖励。
- PostgreSQL 只保存平台数据；高频世界模拟不写入中央数据库。
- PROJECT GSS0 以 30 Hz 在单进程内完成权威共享世界模拟，首次加入返回可读 JSON 状态，运行中以 12 Hz 广播版本化二进制快照；客户端仅上报递增序号和目标方向，通过延迟补间恢复 60 FPS 画面。单人时波次、人机走位、模块、碰撞、特效与声音保持原版规则；多人波次按存活玩家数倍增，每份敌人按对应玩家等级生成。玩家技能只攻击敌蛇；玩家之间仅处理头部反弹和头撞身体死亡。暂停或选择模块时单人世界整体冻结，多人世界仅冻结、保护并闪烁当前玩家。

## 扩展策略

新增游戏只需提供清单、启动地址和服务令牌，并复用 client/server SDK。纯静态游戏可以先声明 `wallet: none`；需要积分时增加一个可信服务端适配器，无需改动大厅。
